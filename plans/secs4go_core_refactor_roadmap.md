# Secs4Go 核心库重构路线图（A：定全局）

## 1. 目标

本路线图只聚焦 `secs4go/` 核心库，目标不是“继续堆功能”，而是把核心库收敛到下面这句话：

**用简洁精悍的结构，提供稳定、可验证、边界清晰的 SECS/GEM 核心能力。**

本轮路线图默认：
- transport 生命周期问题不作为主战场；
- `example/` 暂不优先重构，只作为联调生态层；
- 优先处理核心正确性、模块边界、API 一致性、工程可验证性。

## 2. 重构的全局判断

当前项目不是“方向错了”，而是处在 **协议库雏形已成，但应用层边界尚未收紧** 的状态。

最关键的矛盾有 4 个：
1. `SecsGem` 承担职责过多；
2. reply 模型存在正确性风险；
3. `Message` / `Config` / `Codec` 的抽象仍有运行时耦合和弱约束；
4. 没有测试基线，导致“稳定”难以被证明。

所以全局上不应先做“大重写”，而应走：

**先修正确性 → 再收边界 → 再统一抽象 → 最后补体验与文档。**

## 3. 目标结构

理想状态下，`secs4go/` 应收敛为 4 层：

1. **传输层**：`HSMSTransport`
   - 负责连接、收发、控制会话、状态通知；
   - 不负责应用层 reply 语义。

2. **协议对象层**：`HSMSHeader` / `Message` / `Item`
   - 负责协议对象表达；
   - 不反向依赖 transport 运行时对象。

3. **编解码层**：`ReadHSMSFrame` / `ItemCodec`
   - 负责字节与对象转换；
   - 对错误显式失败，不做静默降级。

4. **应用会话层**：`SecsGem`
   - 负责请求-回复关联、上行回调、对外友好 API；
   - 不偷偷接管 transport 内部状态和重副作用默认值。

## 4. 重构原则

### 4.1 单向依赖
- 允许 `SecsGem -> HSMSTransport`；
- 不鼓励 `Message -> HSMSTransport` 这种反向嵌入；
- 协议对象尽量保持可离线构造、可单测、可回放。

### 4.2 正确性优先于优雅
- 先修 reply 语义和消息路由正确性；
- 再谈接口是否更漂亮。

### 4.3 显式优于隐式
- 显式校验配置；
- 显式处理编码错误；
- 显式选择文件日志；
- 显式声明谁拥有生命周期。

### 4.4 小步收敛，不做爆破式重写
- 每一阶段都应可独立验收；
- 每一阶段都尽量保留现有外部使用方式；
- 先加测试兜底，再动高风险边界。

## 5. 分阶段路线图

### Phase 0：建立最小验证基线

**目标**：让后续重构可回归、可止损。

**要做的事**：
- 为 `secsgem`、`message`、`secs_item_codec` 建最小测试集；
- 覆盖以下最关键行为：
  - `Send()` 请求/回复链路；
  - 回复超时（T3）；
  - `ParseMessage()` 基础解析；
  - 编码配置与异常输入。

**产出**：
- 最小单元测试基线；
- 后续每个阶段都能回归验证。

**验收点**：
- 修改 reply/codec 行为时，能第一时间暴露回归问题。

---

### Phase 1：修正核心正确性问题

**目标**：先让 `SecsGem` 的消息语义可信。

**要做的事**：
- 修复 `Send()` 使用“请求头解析回复”的问题；
- reply 等待结构从“只存 `itemData`”升级为“存真实回复头 + 回复体”；
- 修复“仅以 `WBit=false` 判定回复”的路由问题，避免把无回复主消息误吞为 reply。

**产出**：
- `Send()` 返回的 `Message` 具有真实 reply 语义；
- 非 WBit 主消息也能正常进入上层回调。

**验收点**：
- reply 的 `Stream/Function/WBit/SystemBytes` 与实际收到帧一致；
- 非 reply 的 `WBit=false` 数据消息不会被吞掉。

---

### Phase 2：收紧 `SecsGem` 与 `HSMSTransport` 的边界

**目标**：让“谁负责什么”变清楚。

**要做的事**：
- 明确 transport 生命周期所有权；
- 避免 `NewSecsGem()` 直接修改 transport 内部字段；
- 明确 logger 注入方式；
- 把 `SecsGem` 聚焦到“会话 API + 关联 + 分发”。

**产出**：
- `SecsGem` 不再像“运行时粘合器”；
- `HSMSTransport` 也不再被门面层偷偷改内部状态。

**验收点**：
- 构造链清晰；
- 关闭/启动/注入职责描述一致；
- 边界文档可一段话说清楚。

---

### Phase 3：净化协议对象与配置入口

**目标**：让协议对象“更纯”，让配置“更强约束”。

**要做的事**：
- 减少或移除 `Message` 对 `HSMSTransport` 的直接感知；
- 让 `Config.Validate()` 成为初始化主流程的一部分；
- 明确 `ItemAEncoding` 的合法集合；
- 未知编码和解码失败尽量显式报错。

**产出**：
- `Message` 更适合独立测试和离线构造；
- 配置错误更早暴露。

**验收点**：
- 非法配置不会“悄悄带病启动”；
- 协议对象层不再明显反向依赖 transport。

---

### Phase 4：统一 `Item` / `Logger` 的抽象体验

**目标**：减少 API 的“半成品感”。

**要做的事**：
- 给 `Item` 增加更清晰的 typed accessor 或辅助读取方法；
- 收敛 ASCII/string 的读写不对称；
- 降低默认 logger 的副作用；
- 如保留 file logger，则补齐资源释放语义。

**产出**：
- `Item` 读取更稳定；
- logger 更像库级设施，而不是示例型实现。

**验收点**：
- API 使用者不需要大量依赖裸类型断言；
- 默认行为更轻、更可控。

---

### Phase 5：补文档与生态对齐

**目标**：把“好结构”变成“可交付”。

**要做的事**：
- 增加根级 README；
- 明确支持范围、设计边界、推荐用法；
- 最后再回头收 `example/`，让它反映新 API。

**产出**：
- 使用说明；
- 边界说明；
- 生态样例与核心 API 一致。

**验收点**：
- 新人可通过 README 理解分层与入口；
- example 不再与核心库设计方向打架。

## 6. 推荐推进顺序

建议按下面顺序推进，而不是并行乱改：

1. **测试基线**
2. **reply 正确性**
3. **消息路由正确性**
4. **`SecsGem` / `Transport` 边界收紧**
5. **配置与编码约束**
6. **Item / Logger 一致性**
7. **README 与 example 对齐**

## 7. 本路线图的取舍

这份路线图明确不建议当前就做的事：
- 不建议立刻大规模拆文件；
- 不建议先改 example 层；
- 不建议先做“更漂亮的 API”而忽略 reply 正确性；
- 不建议在没有测试基线时直接大动 `Message` 和 `Item` 抽象。

## 8. 全局结论

对这个项目来说，最重要的不是“再多支持几个消息”，而是先把核心库从“能跑”推进到“语义正确、边界稳定、可回归验证”。

一句话总结：

**全局上应先把 `SecsGem` 从过重的粘合层收缩成真正的应用会话层，再逐步净化 `Message`、`Config`、`Codec`、`Logger` 的边界与约束。**
