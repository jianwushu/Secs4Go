# Secs4Go 最小改造优先级清单（C：定局部）

## 1. 使用方式

这份清单不是“大重构列表”，而是基于当前代码形态，挑出 **最小改动、最大收益、最低扩散风险** 的局部动作。

判断标准只有 3 条：
- 是否直接影响正确性；
- 是否会误导上层 API 使用者；
- 是否能在不大改架构的前提下先落地。

## 2. P0：必须最先动的点

### P0-1 修复 `Send()` 的 reply 解析错误

**位置**：`secs4go/secsgem.go`

**当前问题**：
- reply 等待链路只保存 `itemData`；
- `Send()` 解析 reply 时复用了请求 header；
- 返回给上层的 reply `Message` 头语义可能失真。

**建议动作**：
- 把等待结果从 `[]byte` 提升为“`HSMSHeader + []byte`”；
- `Send()` 用真实 reply header 调用 `ParseMessage()`。

**收益**：
- 直接修正应用层最核心的回复语义问题。

**风险**：
- 局部逻辑变动不大，但会影响 reply 相关测试和既有假设。

**验收点**：
- reply 的 `Stream/Function/WBit/SystemBytes` 与线上的实际回复一致。

---

### P0-2 修复“以 `WBit=false` 判定 reply”的误路由

**位置**：`secs4go/secsgem.go`

**当前问题**：
- `handleDataMessage()` 当前只要 `!msg.WBit` 就走 `handleReply()`；
- 这会把“无需回复的主消息”也当成 reply；
- 如果没有匹配的 pending request，这类消息会被直接吞掉。

**建议动作**：
- reply 判定不能只看 `WBit`；
- 应优先看 `SystemBytes` 是否存在 pending request；
- 若不存在 pending request，则按正常上行消息交给 `OnMessage()`。

**收益**：
- 修复消息吞没风险；
- 修复主消息/回复消息边界错误。

**风险**：
- 低；属于非常值得尽快修正的逻辑错误。

**验收点**：
- 一个 `WBit=false` 的主消息在无 pending request 时仍能被应用层收到。

## 3. P1：低风险高收益的边界收紧

### P1-1 默认 logger 改为轻量默认值

**位置**：`secs4go/secsgem.go`

**当前问题**：
- `logger=nil` 时默认走 `NewFileLogger(deviceName)`；
- 这会在工作目录创建 `logs/`，对库来说副作用偏重。

**建议动作**：
- 默认 logger 改为 `NewDefaultLogger()`；
- 文件日志改为显式选择。

**收益**：
- 立即降低默认副作用；
- 更符合通用库默认行为。

**风险**：
- 会改变现有默认日志落盘行为；
- 但这是合理且可控的行为变化。

---

### P1-2 配置校验前置到启动主流程

**位置**：`secs4go/hsms_transport.go` / `secs4go/config.go`

**当前问题**：
- `Config.Validate()` 存在，但当前流程里不是强制入口；
- 可能带着无效配置继续运行。

**建议动作**：
- 在 `Start()` 入口统一执行 `config.Validate()`；
- 对 `ItemAEncoding` 增加合法值校验。

**收益**：
- 让配置错误尽早暴露；
- 避免后期以奇怪运行时行为暴雷。

**风险**：
- 某些过去“能带病跑起来”的配置现在会直接失败；
- 但这是正确方向。

---

### P1-3 `NewItemCodec()` 对未知编码显式报错

**位置**：`secs4go/secs_item_codec.go`

**当前问题**：
- 未知编码会静默退回 ASCII/UTF-8 路径；
- 调用方很难知道自己配置错了。

**建议动作**：
- 未知编码直接返回错误；
- 去掉“悄悄兜底”的默认分支。

**收益**：
- 让 API 语义真实；
- 减少隐藏问题。

**风险**：
- 会暴露已有错误配置；
- 但这是好事。

## 4. P2：应跟进，但可放在第二轮

### P2-1 减少 `Message` 对 transport 的嵌入

**位置**：`secs4go/message.go` / `secs4go/secsgem.go`

**当前问题**：
- `Message` 持有 `sender *HSMSTransport`；
- 协议对象层被运行时依赖侵入。

**建议动作**：
- 第一阶段先减少直接依赖面；
- 第二阶段再考虑把回复能力从 `Message` 上拆回会话层。

**为什么不放 P0/P1**：
- 这是结构优化，不是最短路径正确性修复；
- 改动扩散面会比 reply 修复更大。

---

### P2-2 减少 `logReceivedData()` 的再编码依赖

**位置**：`secs4go/secsgem.go`

**当前问题**：
- 接收日志会对已解析 `Item` 再编码一次；
- 日志路径和主流程耦合偏深。

**建议动作**：
- 先让日志更多基于已接收 header / item 展示；
- 不在日志路径里重复做协议构造。

**为什么不放更前面**：
- 它主要是设计洁净度问题，不是第一优先级正确性问题。

---

### P2-3 为 `Item` 增加读取辅助，而不是立刻大改 `Value interface{}`

**位置**：`secs4go/item.go`

**当前问题**：
- `Item.Value interface{}` 造成使用者容易写出裸断言；
- 但直接大改 `Item` 结构风险太高。

**建议动作**：
- 先补 typed accessor / helper；
- 暂不立即重构 `Item` 主体模型。

**为什么不立刻大改**：
- 这是 API 级结构问题，牵一发动全身；
- 当前更应该先修 `SecsGem` 的正确性和边界。

## 5. 推荐的最小实施顺序

如果只允许做一轮非常克制的改动，建议顺序如下：

1. `P0-1`：修 reply header/回复体解析
2. `P0-2`：修 reply 误路由
3. `P1-2`：配置校验前置
4. `P1-3`：未知编码显式报错
5. `P1-1`：默认 logger 轻量化
6. `P2-2`：减少日志再编码
7. `P2-1`：逐步净化 `Message`
8. `P2-3`：补 `Item` 读取辅助

## 6. 当前不建议马上动的点

下面这些事现在不建议抢跑：
- 先拆很多文件；
- 先重写 `example/`；
- 先重构 `Item` 整体建模；
- 先做外观层面的 API 美化。

原因很简单：

**当前最紧要的是把核心错误语义修掉，而不是让结构看起来更漂亮。**

## 7. 局部结论

如果只看“最小改造”，那真正该先动的不是大范围重构，而是两刀：

1. **让 reply 解析基于真实回复头；**
2. **让 `WBit=false` 的主消息不再被误吞。**

这两件事修完，核心库的可信度会立刻提升一截；其余边界优化，再顺着全局路线图继续推进。
